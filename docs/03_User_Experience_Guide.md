# C3 BASIC COMPUTER

## 03 User Experience Guide

### Purpose

Every interaction should encourage curiosity.

The computer should feel calm, predictable and approachable.

The experience should invite makers to explore.

---

# First Impression

Power on.

The computer should be ready within seconds.

The first screen is simple.

```text
C3 BASIC COMPUTER

READY.
```

Nothing else is required before creating.

---

# Programming First

Programming is the primary activity.

The computer should never place unnecessary steps between the maker and creating.

---

# Tone

The system speaks clearly.

The system speaks briefly.

The system speaks honestly.

Avoid unnecessary words.

Avoid exaggerated enthusiasm.

Avoid sarcasm.

Avoid judgement.

---

# READY.

READY. is home.

After completing an operation, return to READY.

The prompt should always feel familiar.

---

# HELP

HELP is part of exploration.

Every command should include:

* Description
* Example
* Related commands

Examples are more important than long explanations.

---

# ERROR

Errors are part of exploration.

An error message should explain:

* What happened
* Where it happened
* Nothing more

Example

```text
ERROR

Syntax error.

Line 120.
```

Avoid numeric error codes whenever practical.

---

# WARNING

Warnings should appear only when an action may permanently remove user data.

Warnings should never become routine.

Example

```text
WARNING

This operation removes
all user programs.

Do you understand? (Y/N)
```

Second confirmation

```text
Ready to continue? (Y/N)
```

---

# Success

Success should be quiet.

Whenever practical:

Return to

```text
READY.
```

---

# Recovery

Exploration should never permanently damage the computer.

The user workspace can always be renewed.

Recovery should be simple.

Recovery should never feel like failure.

---

# Consistency

Every command should feel like it belongs to the same computer.

The same concepts should use the same words.

The same actions should produce similar responses.

---

# Simplicity

Prefer simple commands.

Prefer simple messages.

Prefer simple examples.

Small systems are easier to understand.

---

# Curiosity

The computer should encourage curiosity.

Not by talking.

Not by teaching.

But by making exploration easy.

---

# The Goal

The maker should want to turn the computer on.

Not because they have to.

Because they want to see what they can build today.

Keep Going.
